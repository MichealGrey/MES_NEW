# MES 生产管理菜单改善方案

> 基于 `当前MES问题Production.md` 中的问题清单，结合实际代码分析，给出系统性改善方案。

---

## 一、问题诊断总结

### 1.1 代码层面核心问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **Mock 数据泛滥** | `LotListViewModel`、`WipOverviewViewModel`、`LotHoldViewModel` 全部使用硬编码 Mock 数据 | 页面无法反映真实生产状态，无法用于实际业务 |
| **权限体系缺失** | `ISessionService` 仅有 `IsLoggedIn`/`DisplayName`/`EmployeeId`，无角色/权限概念；登录无密码验证 | 任何人都能执行所有操作，不符合 MES 安全要求 |
| **Assemble/Test 未分离** | 所有页面统一展示，无工艺阶段区分 | 前道(晶圆/CP)和后道(封装/FT)混在一起，不符合半导体工厂实际 |
| **页面间无联动** | 各 ViewModel 独立运作，无事件总线或共享状态 | 工单→批次→进站→Hold→拆批 等业务流断裂 |
| **Admin 功能混入生产菜单** | 系统监控、系统健康、外部系统接口与生产操作页面平级 | 普通操作员看到不该看到的内容，权限混乱 |

### 1.2 业务层面核心问题

| 问题 | 说明 |
|------|------|
| **WIP 指标无优先级** | 当前 WIP 页面只显示工序名称和数量，无 Urgent 工单标识、达成率、交期风险 |
| **进站/出站无自动 Hold** | 缺少基于业务规则的自动拦截（良率低于阈值、超重工次数、设备 Down 等） |
| **Hold 维度不精确** | 无法区分是 Hold 整个工单还是单个 Lot，无法按 Package/Bin 精确 Hold |
| **拆批/合批无规则** | 没有定义什么条件下可以拆批/合批，拆合批后追溯链不完整 |
| **良率报表 vs 良率管理重复** | `YieldReportView` 和 `YieldTrendView`/`WaferMapView` 功能重叠 |

## 二、半导体封装厂业务定义

### 2.1 批次层级定义（Lot Hierarchy）

```

│ 层级 1: Wafer Lot（来料晶圆批次）                                                        │
│   来源: 晶圆厂来料 / 客户来料 / 委外回料                                                  │

│   属性: Wafer ID、晶圆厂、晶圆尺寸、Die 数量(Wafer Map)、来料检验结果                     │

│                                                                                         │
│   ↓ 根据工单创建生产批次（1 片 Wafer = 1 个 Mother Lot）                                 │

### 2.2 Split/Merge 规则定义

#### 2.2.1 Split（拆批）规则

| 场景 | 是否允许 | 规则说明 |
|------|---------|---------|
| Wafer Lot → Mother Lot | ✅ 必须 | 来料晶圆按片拆分为 Mother Lot，1 Wafer = 1 Mother Lot。拆分时绑定 Wafer ID，记录 Die 数量（从 Wafer Map 读取） |
| Mother Lot → Sub Lot (Assemble→Test 交接) | ✅ 允许 | Assemble 完成后进入 Test 前，可按测试需求拆分。拆分后 Sub Lot 继承 Mother Lot 的 RouteId，但切换到 Test Route。需要记录 Assemble→Test 的交接信息 |
| 工序间拆分 (Mother Lot 内) | ✅ 允许 | 在工序出站后（Status=Waiting）可按数量拆分。条件: 批次状态为 Waiting/Released，不在 Processing 中。子批继承母批的 RouteId、CurrentStep、所有属性。子批 LotId = 母批 LotId + "-S" + 序号 |
| Grade Split (按 Bin 等级拆分) | ✅ 允许 | Test 完成后按 Bin 结果拆分为不同等级。条件: 批次已完成所有 Test 工序，有 Bin 结果。子批 LotId = 母批 LotId + "-G" + Grade(A/B/C)。拆分后母批状态变为 Completed，子批状态为 Waiting |
| Rework Split (不良品重工拆分) | ✅ 允许 | 发现不良品需要重工时，将不良部分拆出。条件: 批次在工序出站后，有明确的不良数量。子批进入重工路线(Rework Route)，母批继续正常路线。子批 LotId = 母批 LotId + "-R" + 重工次数 |
| Hold 拆分 (部分 Hold) | ✅ 允许 | 部分批次需要 Hold，部分继续生产时。条件: 批次状态为 Waiting/Released。Hold 部分拆出后状态变为 Hold，剩余部分继续生产 |
| ❌ Processing 中拆分 | ❌ 禁止 | 批次正在工序中加工（Status=Processing）时禁止拆分。原因: 数量不确定，无法保证追溯完整性 |
| ❌ Hold 状态拆分 | ❌ 禁止 | 批次被 Hold 时禁止拆分。原因: Hold 原因未解决前，不应改变批次结构 |
| ❌ Completed 后拆分 | ❌ 禁止 | 批次已完成所有工序后禁止拆分（Grade Split 除外）。原因: 成品批次结构已固定 |
| ❌ Scrapped 后拆分 | ❌ 禁止 | 已报废批次禁止任何操作 |

#### 2.2.2 Merge（合批）规则

| 场景 | 是否允许 | 规则说明 |
|------|---------|---------|
| 同工序合批 (Mother Lot 间) | ✅ 允许 | 同产品、同 Route、同工序、同状态的批次可以合并。条件: ProductId 相同、RouteId 相同、CurrentStepSeq 相同、Status 都是 Waiting、合批后数量不超载具容量。合并后保留所有源批次的追溯链 |
| Sub Lot 合批 (Test 阶段) | ✅ 允许 | Test 阶段同 Bin 等级的 Sub Lot 可以合并。条件: ProductId 相同、RouteId 相同、CurrentStepSeq 相同、BinResult 相同、Status 都是 Waiting |
| Rework 后合批 | ✅ 允许 | 重工完成的批次回到原工序后，可以与同工序批次合并。条件: ProductId 相同、RouteId 相同、CurrentStepSeq 相同、Status 都是 Waiting |
| ❌ 不同产品合批 | ❌ 禁止 | 不同 ProductId 的批次禁止合并。原因: 产品不同，无法保证质量追溯 |
| ❌ 不同 Route 合批 | ❌ 禁止 | 不同 RouteId 的批次禁止合并。原因: 工艺路线不同，后续工序无法统一 |
| ❌ 不同工序合批 | ❌ 禁止 | 不同 CurrentStepSeq 的批次禁止合并。原因: 工序进度不同，无法统一跟踪 |
| ❌ 不同 Bin 等级合批 | ❌ 禁止 | 不同 BinResult/Grade 的批次禁止合并。原因: 品质等级不同，混批会导致品质问题 |
| ❌ Hold 批次合批 | ❌ 禁止 | 被 Hold 的批次禁止参与合批。原因: Hold 原因未解决，不应与其他批次混合 |
| ❌ 超载具容量合批 | ❌ 禁止 | 合批后总数量超过载具最大容量时禁止合并。原因: 物理限制，载具无法容纳 |

### 2.3 Hold 维度定义

Hold 维度分类（按 Hold 范围从大到小）：

| # | Hold 维度 | 范围 | 触发场景 | 影响 | 释放条件 | 权限 | 数据库字段 |
|---|----------|------|---------|------|---------|------|-----------|
| 1 | PO Hold（工单级） | 整个工单下所有批次 | 客户暂停、设计变更、物料问题、优先级调整 | 该工单下所有批次全部 Hold | 工单级别释放，所有批次自动恢复 | 生产主管/工程师 | `HoldScope="WorkOrder"`, `ScopeId=OrderId` |
| 2 | Wafer Lot Hold（来料级） | 同 WaferLotId 下所有批次 | 来料质量异常、晶圆厂通知问题、同批次多 Lot 相同异常 | 该 WaferLotId 下所有 Mother Lot 全部 Hold，新创建的也自动 Hold | IQC/品质工程师确认 | 品质工程师/主管 | `HoldScope="WaferLot"`, `ScopeId=WaferLotId` |
| 3 | Device Hold（产品级） | 特定 ProductId 所有在制批次 | 产品设计缺陷、客户投诉、可靠性测试异常、原材料变更 | 该 ProductId 下所有在制批次全部 Hold | 工程变更确认（ECO 完成）后释放 | 工程主管/品质主管 | `HoldScope="Product"`, `ScopeId=ProductId` |
| 4 | Package Hold（封装级） | 特定 PackageType 所有在制批次 | 封装材料异常、工艺参数异常、可靠性测试失败 | 该 PackageType 下所有在制批次全部 Hold | 封装工艺工程师确认 | 工艺工程师/主管 | `HoldScope="Package"`, `ScopeId=PackageType` |
| 5 | Equipment Hold（设备级） | 特定设备上正在加工或等待的所有批次 | 设备 Down/故障、参数偏移(SPC告警)、PM维护、Recipe验证失败 | 该设备 Processing 批次自动 Hold，Waiting 批次禁止进站 | 设备恢复运行、参数验证通过 | 设备工程师 | `HoldScope="Equipment"`, `ScopeId=EquipmentId` |
| 6 | Step Hold（工序级） | 特定 StepCode 所有在制批次 | 工序良率持续偏低、参数异常、操作人员资质问题 | 该工序所有在制批次 Hold，禁止新批次进站 | 工序验证通过后释放 | 工艺工程师/主管 | `HoldScope="Step"`, `ScopeId=StepCode` |
| 7 | Bin Hold（Bin 结果级） | 特定 BinResult 所有批次 | 某 Bin 码系统性问题、客户投诉特定等级、Bin 分选设备异常 | 该 BinResult 下所有批次 Hold | 测试工程师确认 | 测试/品质工程师 | `HoldScope="Bin"`, `ScopeId=BinResult` |
| 8 | Lot Hold（单批次级） | 单个 LotId | 工序良率低于阈值、外观检验异常、测试数据异常、重工超限 | 仅该批次 Hold | 根据 Hold 类型自动/手动释放 | 操作员发起，工程师/主管释放 | `HoldScope="Lot"`, `ScopeId=LotId` |
| 9 | Route Hold（工艺路线级） | 特定 RouteId 所有在制批次 | 工艺路线设计变更、路线验证失败 | 使用该 RouteId 的所有批次 Hold | 工艺路线验证通过后释放 | 工艺工程主管 | `HoldScope="Route"`, `ScopeId=RouteId` |

**Hold 维度优先级**（当多个 Hold 同时存在时）：

```
PO Hold > Wafer Lot Hold > Device Hold > Package Hold > Route Hold > Equipment Hold > Step Hold > Bin Hold > Lot Hold
```

即：范围越大的 Hold 优先级越高，必须先释放大范围 Hold 才能释放小范围 Hold

### 2.4 MES 生命周期边界定义

```
MES 生命周期：从来料到成品仓入库

阶段 1: 来料管理（Incoming）
  晶圆来料登记 → IQC 检验 → Wafer Map 导入 → 生成 Wafer Lot 编号 → 入库待产
  MES 关注点: 来料批次号、晶圆厂、晶圆 ID、Die 数量、来料检验结果、Wafer Map
  结束标志: Wafer Lot 状态变为 "Released to Production"

  ↓ 根据工单创建生产批次

阶段 2: 前道生产（Assemble / Front-End）
  DieAttach → WireBond → Mold → Cure → TrimForm → Marking → LaserMark → VisualInspection
  MES 关注点: 每道工序的进出站、数量、良率、设备、载具、Recipe、操作员
  可执行操作: Split（工序间）、Merge（同工序）、Hold（任意时刻）、Rework
  结束标志: 所有 Assemble 工序完成，批次状态变为 "Assemble Completed"

  ↓ Assemble/Test 交接（可能再次拆分）

阶段 3: 后道生产（Test / Back-End）
  CP Test → FT Test → BurnIn → FinalInspection → Packing
  MES 关注点: 测试数据、Bin 结果、良率、测试程序版本
  可执行操作: Split（按 Bin 等级）、Merge（同 Bin 等级）、Hold、Rework
  结束标志: 所有 Test 工序完成，批次状态变为 "Test Completed"

  ↓ 按 Bin 等级拆分（Grade Split）

阶段 4: 成品管理（Finished Goods）
  Grade Split → 生成 Grade Lot → 成品检验 → 成品入库 → 生成入库单
  MES 关注点: 等级分布（A/B/C 各多少）、最终良率、入库数量、入库时间
  结束标志: 成品仓入库完成，批次状态变为 "InWarehouse" → "Shipped"

  ⚠️ MES 生命周期终点: 成品仓入库完成（Shipped）
     Shipped 后 MES 不再跟踪该批次，后续由 ERP/WMS 管理
```

**生命周期状态机：**

```
Created → Waiting → Processing → (循环: Processing ↔ Waiting) → Completed → InWarehouse → Shipped (MES 结束)
                 ↕
               Hold → Released → Waiting
                 ↕
               Scrapped (异常终止)
```

## 三、整体架构改善

### 2.1 菜单结构重组

```
MES 主菜单
├── 📋 生产执行 (Production Execution)          ← 操作员/班组长
│   ├── 工单管理        (WorkOrderListView)
│   ├── 批次管理        (LotListView)            ← 需重写
│   ├── WIP 总览        (WipOverviewView)        ← 需重写
│   ├── 进站/出站       (TrackInView)
│   ├── 批次 Hold       (LotHoldView)
│   └── 拆批/合批       (LotSplitMergeView)
│
├── 📊 生产监控 (Production Monitoring)          ← 工程师/主管
│   ├── 主数据管理      (MasterDataView)
│   ├── 客户交付跟踪    (CustomerProgressView)
│   ├── 生产报表        (ProductionReportView)
│   └── 良率管理        (YieldDashboardView)     ← 合并 YieldReport + YieldTrend + WaferMap

    ├── 用户管理
    ├── 角色管理

public class AutoHoldRuleEngine
{
    /// 出站时自动检查，触发 Hold 则禁止出站
    public async Task<AutoHoldResult> CheckBeforeTrackOutAsync(LotInfo lot, TrackOutRequest request)
    {
        var checks = new List<AutoHoldCheck>
        {
            // 1. 良率低于阈值自动 Hold
            new YieldThresholdCheck { Threshold = 0.95 },

            // 2. 重工次数超限自动 Hold
            new ReworkLimitCheck { MaxReworkCount = 3 },

            // 3. 设备 Down 时禁止进站
            new EquipmentStatusCheck { AllowedStatuses = ["Idle", "Running"] },

            // 4. 连续不良品超限自动 Hold
            new ConsecutiveFailCheck { MaxConsecutiveFails = 5 },

            // 5. 超期未完成自动 Hold
            new OverdueCheck { MaxHoursInStep = 48 },

            // 6. 载具不匹配自动 Hold
            new CarrierMismatchCheck(),

            // 7. Recipe 未验证自动 Hold
            new RecipeValidationCheck()
        };

        foreach (var check in checks)
        {
            var result = await check.ExecuteAsync(lot, request);
            if (result.Triggered)
                return AutoHoldResult.Hold(result.HoldType, result.Reason);
        }

        return AutoHoldResult.Pass();
    }
}

### 3.7 拆批/合批

**问题：** 无规则定义、无 Assemble/Test 区分

**方案：**

#### 3.7.1 拆批规则

```
允许拆批的条件：
1. Lot 状态为 Waiting 或 Released（非 Hold/Completed/Scrapped）
2. 不在进行中（无 TrackIn 记录且未 TrackOut）
3. 操作员有 Lot.Split 权限

拆批方式：
- 按数量拆分：指定子批数量，剩余留在母批
- 按比例拆分：按百分比拆分
- 按载具拆分：不同载具的单元拆为独立 Lot

拆批后：
- 子批 LotId = 母批 LotId + "-S1", "-S2" ...
- 继承母批的 RouteId、CurrentStep、所有历史记录
- MotherLotId 指向母批，可追溯
```

#### 3.7.2 合批规则

```
允许合批的条件：
1. 所有 Lot 的 ProductId 相同
2. 所有 Lot 的 RouteId 相同
3. 所有 Lot 的 CurrentStep 相同
4. 所有 Lot 状态为 Waiting
5. 合批后总数量不超过载具容量

合批后：
- 新 LotId = 合并时间戳 + 序号
- 保留所有源 Lot 的追溯链
- SourceLotIds 记录来源
```

    Task LogoutAsync();

// 2. ViewModel 基类权限检查
public abstract class AuthorizedViewModel : BindableBase
{
    protected readonly IAuthorizationService _auth;

    protected bool CanExecute(string module, string action)
        => _auth.HasPermission(module, action);

    protected async Task ExecuteWithPermissionAsync(
        string module, string action, Func<Task> action)
    {
        if (!_auth.HasPermission(module, action))
        {
            StatusMessage = $"无权限执行此操作 ({module}.{action})";
            return;
        }
        await action();
    }
}

// 3. TrackInViewModel 集成权限
public class TrackInViewModel : AuthorizedViewModel
{
    private async void OnConfirm()
    {
        await ExecuteWithPermissionAsync("Track", "Confirm", async () =>
        {
            // 原有确认逻辑
            await _trackService.TrackInAsync(...);
        });
    }
}
